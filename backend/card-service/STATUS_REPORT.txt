🏗️  ARCHITECTURE OVERVIEW
═══════════════════════════════════════════════════════════════════

User Request
    ↓
CardPaymentService (Entry Point)
    ├─ Validates request
    ├─ Checks idempotency
    └─ Delegates to MerchantPaymentService
        ↓
    MerchantPaymentService
        ├─ Retrieves card
        ├─ Creates payment flow
        └─ Delegates to flow
            ↓
        PaymentFlowFactory
            ├─ CREDIT → CreditCardPaymentFlow
            └─ DEBIT → DebitCardPaymentFlow
                ↓
            Selected Flow
                ├─ Validates IBAN (format, checksum, length)
                ├─ Validates CVV (numeric, 3-4 digits)
                ├─ Validates Expiry (format, not expired)
                ├─ Validates Card (status, not blocked)
                ├─ Processes through TransactionGateway
                ├─ Creates CardTransaction
                ├─ Persists to database
                ├─ Applies charges (credit only)
                └─ Returns response
                    ↓
                Response with:
                • Reference number
                • Transaction status
                • Amount charged
                • Idempotency preserved

🎓 SOLID PRINCIPLES IMPLEMENTATION
═══════════════════════════════════════════════════════════════════

S - Single Responsibility ✅
  Each class has ONE reason to change
  • IbanValidator → IBAN validation only
  • CvvValidator → CVV validation only
  • CreditCardPaymentFlow → Credit processing only
  • DebitCardPaymentFlow → Debit processing only

O - Open/Closed ✅
  Open for extension, closed for modification
  • Add new payment flows without changing code
  • Factory pattern handles routing
  • Implement PaymentFlow interface = done!

L - Liskov Substitution ✅
  Subtypes are substitutable for base types
  • CreditCardPaymentFlow implements PaymentFlow
  • DebitCardPaymentFlow implements PaymentFlow
  • Both interchangeable

I - Interface Segregation ✅
  Small focused interfaces
  • PaymentFlow: 2 methods only
  • Each validator has specific purpose
  • No fat interfaces

D - Dependency Inversion ✅
  Depend on abstractions, not concretions
  • MerchantPaymentService → PaymentFlow (abstraction)
  • Not → CreditCardPaymentFlow (concrete)
  • Factory handles creation
